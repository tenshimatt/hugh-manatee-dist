## Project anatomy (Chris's words, tidied)

Each **project** contains:

- Schedule (guide/Gantt chart — templated, PMs tweak, formula-driven RAG status)
- Budget
- Change Order Request log
- Change Order Budget
- Forecast
- Production
- Project Charter (pulls from contract)

Each **HPM** (Head/Head Project Manager?) has their own dashboard; each project has its own dashboard; values roll up. Chris's Excel workbook is the current system of record and he's happy for us to replicate it in the new UI.

Schedules today are **field, shop, engineering** — only *field* is "live"; shop + engineering are forward intent ("what we expect to happen"). **The ERF (Engineering Request Form) is the real live schedule** — PM submits job # (e.g. 26052), release number, parameters; on submit it lands in the **Production Sheet**.

### Handover rule (hard constraint)

> "You can't have a card in engineering and in the shop because that means you're making something that hasn't finished."

Sales → Engineering (rubber-stamp if trivial) → Shop Floor. Even a single-step engineering review still has to be stamped. This is the gate Chris wants productionised (approval workflow, email-with-button style OK).

### Engineering stages (canonical list from Chris)

uncategorized → evaluating → float → layout → layout check → sketch → sketch check → correction → CNC programming → laser programming → punch programming → program complete → **release to shop**

"Release to shop" is the one stage that moves the card across the engineering/shop boundary.

### Assembly hierarchy (Processing side)

Parts (produced on tube laser / flat laser / form / weld) → Stock part → take N parts (e.g. 4) → Sub-assembly → Final assembly → Ship.

Chris says this part-of-a-part modelling is something they currently struggle with.

## Router / Route (shop workflow per job)

- **Defined at estimate time.** Bespoke per project but built from a known station library.
- Examples: `laser → press brake → weld` or `laser → weld → press brake`.
- On NCR, the route can: (a) loop back to earlier station OR (b) branch to an optional station (e.g. finishing to remove burrs) and then rejoin.
- Chris explicitly linked this to n8n visual workflow and liked the "A → B → C, D forks" mental model. He wants the same visual on each work package.

> Verbatim: "when they estimate a job, they determine the path it's going to take. That, I guess, is the workflow."

**Matt's commitment on-call:** once rules are explained, the system will be contextually aware — when two machines are busy it should look ahead and "smart distribute" based on what's coming. Chris: *"you just nailed a really important point."*
